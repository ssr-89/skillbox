Что такое Git?

Git — абсолютный лидер по популярности среди современных систем управления версиями. Это развитый проект с активной поддержкой и открытым исходным кодом. Система Git была изначально разработана в 2005 году Линусом Торвальдсом — создателем ядра операционной системы Linux. Git применяется для управления версиями в рамках колоссального количества проектов по разработке ПО, как коммерческих, так и с открытым исходным кодом. Система используется множеством профессиональных разработчиков программного обеспечения. Она превосходно работает под управлением различных операционных систем и может применяться со множеством интегрированных сред разработки (IDE).

Git — система управления версиями с распределенной архитектурой. В отличие от некогда популярных систем вроде CVS и Subversion (SVN), где полная история версий проекта доступна лишь в одном месте, в Git каждая рабочая копия кода сама по себе является репозиторием. Это позволяет всем разработчикам хранить историю изменений в полном объеме.

Разработка в Git ориентирована на обеспечение высокой производительности, безопасности и гибкости распределенной системы.

Часто используемые команды

  • git add . или имя файла — индексирует изменения и даёт возможность откатить изменения.

  • git commit -m "комментарий".

  • git push --set-upstream origin dev или -u origin dev — загрузить на сервер файлы (—set-upstream или -u — параметр на всякий случай, чтобы наш репозиторий на компе был связан с репозиторием на удалённом сервере).

  • git pull origin наименование ветки — команда для дальнейших загрузок с сервера в без клонирования.

  • git checkout -b название ветки — создание ветки и переключение на созданную ветку.

  • git checkout название ветки — переключение между ветками.

Информационные команды

  • git status — показывает статус файлов в папке.

  • ls — посмотреть какие файлы есть в папке, в которой сейчас находимся.

  • ls -la — посмотреть ВСЁ содержимое папки.

  • pwd — показывает в какой папке находимся.

  • git show — показывает о последнем commits и его содержимом.

  • git log — показывает всю историю commits.

  • git blame — показывает автора коммита.

  • git branch — просмотр всех локальных веток в репозитории. Звёздочкой * будет указана текущая ветка.

  • git branch -a — просмотр всех локальных и удалённых веток. Звёздочкой * будет указана текущая ветка.

  • git remote -v - путь к настроенному удалённому репозиторию.

  • git stash list - вывод списка отложенных изменений.

  • git diff - просмотр незакомиченных изменений в проекте.

  • git diff <file> - просмотр незакомиченных изменений в файле.

  • git diff <commit> - просмотр всех изменений между указанным коммитом и текущей версией проекта. Можно указать первые четыре цифры номера коммита.

  • git diff <commit1> <commit2> - просмотр всех изменений между определёнными коммитами, от наиболее раннего к наиболее позднему.

  • git diff <commit1> <commit2> <file> - просмотр изменений в файле между определёнными коммитами, от наиболее раннего к наиболее позднему.

  • git diff <file1> <file2> - просмотр незакомиченных изменений в нескольких файлах.

  • git blame <file> - просмотр изменений в каких коммитах, каким пользователем, когда и какие изменения.

Инициализация локального репозитория и подключение к новому удалённому репозиторию

  • git init - инициализация Git, где нет Git-репозитория. Создаёт папку с файлами Git.

  • Добавить файл в папку проекта.

  • git add . - добавить в индекс файл.

  • git commit -m "комментарий" - добавить в коммит.

  • git remote add origin "ссылка на новый созданный удалённо репозиторий".

  • git remote -v - путь к настроенному удалённому репозиторию.

  • git push -u origin master - добавить в удалённый репозиторий.

Подключение к удалённому репозиторию с помощью клонирования

  • git clone ссылка на репозиторий — копирование файлов на комп с сервера (подключение к удалённому репозиторию).

  • git add . - добавить в индекс файл.

  • git commit -m "комментарий" - добавить в коммит.

  • git push -u origin master - добавить в удалённый репозиторий.

Удаление и изменение пути репозитория

  • git remote remove origin - origin имя удалённого репозитория.

  • git remote add origin "ссылка на новый созданный удалённо репозиторий" - подключение к новому репозиторию.

Ветки

  • git branch - просмотр списка всех веток и текущей ветки, отмеченной *

  • git checkout -b имя ветки - локальное создание ветки и переключение на неё.

    ВАЖНО!!! После создания локальной ветки необходимо закоммитеть, либо спрятать все изменения, чтобы после переключения к другой ветке не произошёл конфликт версий.

  • git branch имя ветки - локальное создание ветки.

    ВАЖНО!!! После создания локальной ветки необходимо закоммитеть, либо спрятать все изменения, чтобы после переключения к другой ветке не произошёл конфликт версий.

  • git checkout имя ветки - переключение на другую ветку.

  • git merge work - слияние веток work -> main. При вводе данной команды необходимо находиться в ветке main.

Конфликт версий

  • git pull - при лёгком варианте конфликта.

  • При сложном варианте - в редакторе кода выбрать лучший вариант, другие удалить и закоммитеть, а далее запушить.

rejected - отклонено

:wq - выход из режима редактирования, например как (END).

Откладывание изменений

  • git stash - сохранение всех незакоммиченных изменений, в том числе, добавленных в индекс, и присваивается статус WIP.

    Выводится сообщение: Saved working directory and index state WIP on basket-delivery-options: b2564a5 changed index.html

  • git stash list - вывод списка отложенных изменений.

  • git stash pop - возвращение последних отложенных изменений.

    После возврата происходит слияние с закоммиченными изменениями.

  • git stash drop 4 - удаление отложенных изменений. 4 - означает номер отложенных изменений, который надо удалить.

Модели Git

Модель Git Flow

  • Develop (dev, devel) - ветка, в которую вливаются все свежие изменения и проверяется общая работоспособность кода.

  • Release - после проверки кода в Develop, из которой вливается в Release.

    Далее тщательно тестируется и вливается в главную ветку Master или Main.

  • Hotfix (hotfixes) - ветка для быстрых изменений.

    Позволяет обходить долгий процесс выкладки изменений.

  • Feature

Модель TBD (Trunk Based Development)

По данной модели создаются отдельные ветки по определённым задачам.

После выполнения задачи ветки вливаются в основную ветку и удаляются.

Просмотр изменений

  • git diff - просмотр незакоммиченных изменений в проекте.

  • git diff <file> - просмотр незакоммиченных изменений в файле.

  • git diff <commit> - просмотр всех изменений между указанным коммитом и текущей версией проекта. Можно указать первые четыре цифры номера коммита.

  • git diff <commit1> <commit2> - просмотр всех изменений между определёнными коммитами, от наиболее раннего к наиболее позднему.

  • git diff <commit1> <commit2> <file> - просмотр изменений в файле между определёнными коммитами, от наиболее раннего к наиболее позднему.

  • git diff <file1> <file2> - просмотр незакоммиченных изменений в нескольких файлах.

Отмена закоммиченных изменений и удаление незакоммиченных изменений

  • git checkout <commit> <file> - отмена закоммиченных изменений и возврат в индекс.

  • git restore --staged <file> - удаление из индекса и возврат в рабочую директорию.

  • git restore <file> - удаление незакоммиченных изменений.

  • git revert <commit> - удаление коммита. Во время удаления будет предложено отредактировать коммит и закоммитеть удаление.

  • CTRL + C - выход из режима редактирования коммита.

  • git revert --no-commit <commit> - удаление коммита без редактирования, сделав один коммит в конце.

Отмена (сброс) коммитов

  • git reset --soft <commit> - мягкий режим с параметром soft.

    Последний коммит будет сброшен до предыдущего и все изменения будут отправлены в индекс.

  • git reset --mixed <commit> или git reset HEAD~1 - смешанный сброс последнего коммита до предыдущего.

    HEAD~1 - цифра означает количество последних коммитов для сброса.

    Все изменения при этом попадают в рабочую директрию без добавления в индекс.

  • git reset --hard <commit> - безвозвратное удаление последующих за ним коммитов. Сам коммит будет последним.

  • git reset --hard HEAD - сбрасывает все изменения в локальном репозитории до последнего коммита и удаляет все незафиксированные изменения.

Вопросы и ответы

  • В какой последовательности при работе с удалённым репозиторием обычно встречаются следующие команды?

    • git clone ... git push ... git pull

    • git clone ... git pull ... git push

  • Вы решаете использовать для своих проектов систему GitLab. Какие последовательности команд возможны при работе в этой системе?

    • git init, git add, git commit, git remote add origin path, git push -u origin master

    • git remote add origin path, git push -u origin master

    • git clone path

    • git init, git remote add origin path, git add, git commit, git push -u origin master

  • С помощью каких команд можно получить свежую версию кода из удалённого репозитория?

    • git pull

    • git clone

  • Чем различаются команды git checkout -b branch-name и git checkout branch-name?

    Параметр -b указывает на необходимость создать ветку и уже потом в неё переключиться.

  • В каком случае необходимо использовать команду git push -u origin branch-name, а в каком достаточно команды git push?

    git push -u origin branch-name выполняется при первой отправке локальной ветки в удалённый репозиторий.

  • Вы последовательно выполняете следующие действия в репозитории:

    1. Изменяете файл 1

    2. Выполняете git stash

    3. Изменяете файл 2

    4. Выполняете git stash

    5. Выполняете git stash pop

    Изменения в каком файле будут возвращены из stash в рабочую директорию?

    В файле 2

  • Что произойдёт, если программист выполнит следующие команды?

    git checkout branch-1

    git merge master

    Ответ: Ветка master будет влита в ветку branch-1.

  • При выполнении каких команд могут возникать локальные конфликты слияния?

    • git merge
    • git pull
    • git stash pop

  • Чем функционально отличается pull request от merge request?

    Ничем.

  • В чём преимущества модели ветвления trunk-based перед Git Flow?

    • Она предполагает более быструю разработку.

    • В ней все ветки, кроме master, «живут» недолго.

    • Она проще для понимания и использования.

    • В ней можно включать и отключать различные фичи, используя feature flags.
